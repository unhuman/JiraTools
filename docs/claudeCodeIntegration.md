# Using JiraTools with Claude Code

Claude Code enables interactive use of JiraTools via MCP (Model Context Protocol) servers, allowing you to leverage AI for complex Jira workflows without manual script execution.

## Prerequisites

- [Claude Code](https://claude.ai/code) CLI or desktop/web app
- MCP server(s) configured in Claude Code:
  - **Jira MCP** — for creating and managing tickets
  - **Backstage MCP** (optional) — for team and component queries
- Jira PAT (Personal Access Token) configured in your MCP server
- Local copy of JiraTools repository (for reference and CSV files)

## Setup

### 1. Configure Jira MCP Server

In Claude Code settings, configure your Jira MCP server with your Jira instance credentials:

```json
{
  "jira_server": "https://jira.cvent.com",
  "personal_access_token": "your-pat-here"
}
```

### 2. Optional: Configure Backstage MCP Server

If using Backstage-dependent workflows (team lookups, component analysis):

```json
{
  "backstageUrl": "https://backstage.core.cvent.org/"
}
```

### 3. Generate Input Data

For ticket creation workflows, use the Python scripts to generate CSV files:

```bash
# Generate standardized ticket CSVs
python standardTicketCreator.py teams.xlsx --csv output.csv

# Generate developer metrics reports
python developerMetrics.py --teams Team-Rocket --period month --filePrefix /tmp/metrics
```

## Common Workflows

### Workflow 1: Bulk Ticket Creation (standardTicketCreator)

Use Claude Code to create multiple tickets from a CSV file with proper field mapping.

**Setup:**
1. Generate per-team CSV files using `standardTicketCreator.py --csv`
2. Open Claude Code and prepare your Excel config file
3. Instruct Claude to read the CSV and create tickets

**Example Claude Code instruction:**

```
Read my CSV file at [path]/Team-Rocket_tickets.csv and my Excel config at [path]/teams.xlsx.

Using the CustomFields sheet, map the following:
- Epic Link → use epicKey field (e.g., {"epicKey": "EPIC-123"})
- Sprint Team → customfield mapping with value wrapper
- Sprint → customfield mapping with numeric sprint ID
- Priority → {"priority": {"name": "4-Medium"}}

For each row in the CSV, create a Jira ticket using jira_create_issue with:
- project_key: from config
- summary: from CSV "Summary" column
- issue_type: from CSV "Issue Type" column
- description: from CSV "Description" column
- assignee: from CSV "Assignee" column
- components: from CSV "Components" column
- additional_fields: JSON with all custom field mappings

Process all rows in parallel where possible. Return the created issue keys.
```

**Expected output:** List of created ticket keys ready for review in Jira.

### Workflow 2: Epic Status Report (epicStatus)

Generate sprint-by-sprint status summaries for an epic.

**Setup:**
1. Identify the epic key (e.g., `PROJ-1000`)
2. Use Claude Code with Jira MCP to query issues

**Example Claude Code instruction:**

```
Using the Jira MCP server, query all issues for epic PROJ-1000:
- Use jira_search with JQL: parent = PROJ-1000 ORDER BY rank
- Return fields: summary, status, assignee, sprint, storyPoints

Group results by sprint and generate a markdown status table showing:
- Sprint name
- Issue count per status (To Do, In Progress, Done)
- Total story points completed this sprint

Format as a markdown report suitable for sharing.
```

### Workflow 3: Developer Metrics Export (developerMetrics)

Generate productivity metrics and visualize them.

**Setup:**
1. Run `developerMetrics.py` locally to generate CSV files
2. Use Claude Code to analyze and present the data

**Example Claude Code instruction:**

```
I have developer metrics CSVs generated by developerMetrics.py:
- Raw issues: [path]/_raw.csv
- Weekly aggregates: [path]/_aggregated.csv

Please:
1. Read both CSV files
2. Calculate team and individual summaries:
   - Total issues completed
   - Average estimate per issue
   - Issues per team
3. Identify top performers and trends
4. Generate a markdown summary with:
   - Team performance table
   - Weekly trend analysis
   - Key observations
```

### Workflow 4: Custom Field Discovery (findCustomFields)

Locate custom field IDs for mapping.

**Setup:**
1. Identify a sample Jira issue containing the custom fields you need
2. Use Claude Code to extract field information

**Example Claude Code instruction:**

```
Using the Jira MCP, read issue PROJ-123 with all fields (*all).

Extract and list all custom fields (customfield_*) with:
- Field ID
- Field name
- Current value
- Field type (if visible)

Format as a markdown table for easy reference in configuration files.
```

## Field Mapping Reference

### Standard Jira Fields

These work with all Jira MCP operations:

```json
{
  "assignee": "user@example.com",
  "description": "Markdown text here",
  "summary": "Issue title",
  "priority": {"name": "4-Medium"},
  "labels": ["tag1", "tag2"],
  "duedate": "2026-05-15"
}
```

### Custom Fields via additional_fields

Custom field mapping depends on your Jira instance. Use `findCustomFields.py` to discover IDs:

```json
{
  "additional_fields": {
    "customfield_10000": "direct value",
    "customfield_10001": {"value": "wrapped value"},
    "customfield_10002": {"id": "option-id"},
    "customfield_10003": ["option1", "option2"]
  }
}
```

### Epic Link Special Handling

Always use `epicKey` field (not raw customfield) to avoid permissions errors:

```json
{
  "additional_fields": {
    "epicKey": "PROJ-456"
  }
}
```

### Sprint Assignment

Sprint requires numeric sprint ID:

```json
{
  "additional_fields": {
    "customfield_10020": 1001
  }
}
```

## Best Practices

### 1. Batch Operations

Use Claude Code's parallel processing for bulk ticket creation:

```
Create 100+ tickets in parallel batches of 10.
Wait for each batch to complete before starting the next.
Collect all created keys and return summary.
```

### 2. Error Handling

Always ask Claude to:
- Validate input data before creation
- Log failed ticket keys separately
- Provide a summary of successes and failures
- Suggest corrections for failed entries

### 3. Configuration Management

Store your custom field mappings in a reusable format:

```yaml
# config/field_mappings.yaml
custom_fields:
  epic_link:
    field: epicKey
    type: epic
  sprint:
    field: customfield_10020
    type: numeric
  sprint_team:
    field: customfield_10019
    type: value_wrapped
```

### 4. Data Validation

Always validate before bulk operations:

```
Before creating tickets:
1. Check that all required fields are present
2. Validate assignees exist in Jira
3. Verify epic keys and sprint IDs are valid
4. Check for duplicate summaries within the project
5. Report validation results before proceeding
```

### 5. Testing

Start with a small batch to verify field mappings:

```
Test with 3 sample tickets first.
Review created tickets in Jira.
Verify all fields (including custom fields) populated correctly.
Then proceed to full batch.
```

## Limitations & Considerations

### What Works Well with Claude Code

- ✅ Ticket creation with complex field mapping
- ✅ Bulk operations with parallelization
- ✅ Data transformation and CSV processing
- ✅ Report generation and analysis
- ✅ Epic and sprint queries via JQL
- ✅ Attachments and remote links

### What's Better as Python Scripts

- ❌ Real-time Backstage team member filtering (use `developerMetrics.py`)
- ❌ Large-scale code audits (use `codeAudit.py` for sparse checkout + regex)
- ❌ Complex service consumer analysis (use `serviceConsumerAnalysis.py` with Datadog)
- ❌ Repeated periodic operations (schedule Python scripts via cron)

### API Rate Limits

- Jira rate limits: ~60 requests/minute (1 per second safe)
- Backstage rate limits: ~100 requests/minute
- When bulk creating 100+ tickets, batch in groups of 5-10 with delays

### Custom Fields & Permissions

- Some custom fields require specific field contexts
- If you get permission errors, try using `epicKey` instead of raw customfield ID
- Verify your Jira user has permission to set each custom field

## Troubleshooting

### "Permission denied" on custom field update

**Solution:** Use the `epicKey` field for epic links instead of raw customfield ID.

### Custom field value not appearing in response

**Solution:** Verify you're using the correct data wrapper format:
- Some fields need `{"value": "..."}` wrapper
- Some need `{"id": "..."}` wrapper
- Some accept raw values directly

Use `findCustomFields.py` to inspect the actual field structure on a sample issue.

### Backstage API calls too slow

**Solution:** Use Claude Code to batch Backstage queries or pre-fetch all team data into a lookup structure before processing.

### Bulk ticket creation times out

**Solution:** Reduce batch size from 10 to 5, or increase delays between batches. For 100+ tickets, expect 5-10 minutes with safe API throttling.
